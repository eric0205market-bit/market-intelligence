# DISCOVERY LAYER — дизайн
## 2026-07-06 · supersedes CONTEXT_Discovery_Loop_Jun10.md · план: BUILD_PLAN_Discovery_Jul06.md

Исходник Jun-10 был гипотезой из двух сообщений. Этот документ — итог отдельного
разбора в Discovery-чате. Слой сквозной: этот чат проектирует и пишет спеки; ветки
и модули строятся в своих чатах.

## Суть
Дискавери — не модуль, а слой поверх RAG + онтологии. Сбор сильный, постановка на
учёт работает (frontier_moat). Слабое звено: система не отличает важное от фонового —
нечем сверять и нечем сравнивать накопленное. Это расширение инвестиционного
процесса владельца: его голова + структурированная память + несколько ЛЛМ-голов,
думающих и проверяющих поверх проверенных источников. Детерминированной метрики
успеха нет и не должно быть — ценность судит владелец как пользователь процесса.

## Две задачи — нужны обе
- Задача 1 — тянуть (pull): копать заявленные темы. Подтверждения, опровержения,
  связи, финансовые следствия.
- Задача 2 — толкать (push): ловить незаявленное (класс компьют-инсайта 2025).
  Вопросами не достаётся — система выносит сама.

## Принцип: один движок, два триггера
Хранилище и поиск одни. Задачу 1 запускает вопрос владельца; задачу 2 — узоры в
данных (повтор, новизна, противоречие). Не две системы — одна труба, два крючка.

## Задача 1 (тянуть) = RAG
Вопрос → эмбеддинг → similarity + фильтр (source/date/entities) → обход онтологии
(причинно связанное, не только похожее) → синтез Opus → разбор. Разовые вопросы +
постоянный список (из агенды), прогоняемый каждым проходом. Может выйти онлайн до
онтологии (сначала по похожести).

## Задача 2 (толкать) = конвейер, не детектор
Три дешёвых фильтра ПАРАЛЛЕЛЬНО, к владельцу напрямую не выходят (шумные) — кормят синтез:
- Новизна: расстояние до соседей > порога → выбросы (нужны ворота важности)
- Повтор: кластеры + счёт НЕЗАВИСИМЫХ источников, растущие → схождения
  (самый перспективный фильтр; тень компьют-инсайта)
- Противоречие: факт vs записанные правила/тезисы → нарушители (самый ценный сигнал,
  требует машиночитаемых ожиданий — часть онтологии)
Сужение → синтез SHIFTS (Opus рассуждает над суженным; агенда пересортировывает,
НЕ режет) → отчёт «на что посмотреть и почему» → человек.
Правило: механика сужает шланг → модель рассуждает → человек решает.
Ручной «прогон Opus по отчётам» без базы = слоп (нет памяти, нет счёта независимости,
нет ожиданий) — НЕ делать.

## Watch-агенда
НЕ поле сбора. Верхний слой, файл в MI-репо: открытые вопросы · активные тезисы ·
экспозиции · слабости. В задаче 1 — сам запрос; в задаче 2 — пересортировка
результата, НЕ фильтр (иначе задача 2 схлопнется в задачу 1).

## Ворота продвижения (человек, при постановке находки на учёт — не фильтр сбора)
измерим · причинная связь · lead/coincident/lag · фальсифицируем · данные доступны

## Для веток сбора — ГЛАВНОЕ
НИЧЕГО дискавери-специфичного в ветки не вносить: ни поля «кандидат», ни агенды.
Строить как YouTube / card schema v1. Причина: «находка или нет» — свойство всей
кучи (повтор/новизна поштучно не видны), поштучный флажок = суждение в изоляции,
потребителя у него нет. Обязательна только дисциплина схемы:
source / date / entities / raw / самодостаточный claim — по ним считаются
похожесть и повтор.

## Граница
Индикатор из личного опыта (friction-index) рождает владелец; система амплифицирует
через повтор (текстовую тень), не заменяет.

## Решено сейчас vs настраивается на живых данных
Сейчас: форма (движок/триггеры/конвейер/агенда-сортирует/ворота).
На потоке: пороги повтора (сколько источников, окно), ворота новизны, такт push
(~2 недели), при требовании: весь ритуал ≤15 мин владельца раз в 1–2 недели.

## Привязка к ARCHITECTURE_v4
Не новый модуль: конкретизация build order п.2 (storage), п.3 (embeddings+ontology),
п.5 (SHIFTS), п.6 (Committee — несколько голов + память решений «почему вошёл, что
должно подтвердиться»).
